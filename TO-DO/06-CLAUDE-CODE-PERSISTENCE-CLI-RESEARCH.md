# SECTION 1: CLAUDE.MD FILES

## 1.1 What is CLAUDE.md?

A plain-text Markdown file containing persistent instructions that Claude reads at the start of every session. Not enforced configuration rules but guidance loaded into the context window as a user message.

**Key characteristics:**
- Written by users/teams (not generated by Claude)
- Loaded into context as a regular message, not system prompt
- Survives `/compact` operations (re-read from disk after compaction)
- Context cost is real — files consume tokens alongside conversation

## 1.2 Multiple CLAUDE.md Files — Hierarchy

### Scope hierarchy (highest to lowest priority):
1. **Managed policy** — macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`, Linux: `/etc/claude-code/CLAUDE.md`, Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`
2. **Project** — `./CLAUDE.md` or `./.claude/CLAUDE.md` (preferred)
3. **User** — `~/.claude/CLAUDE.md`

### Loading order:
1. Walk up directory tree from CWD
2. Load from current directory
3. Load user-level
4. Load managed policy
5. Lazy-load subdirectories when accessed

All matching files are loaded and merged (concatenated). No override mechanism.

## 1.3 Referencing Other Files

Use `@path` import syntax:
```markdown
See @README.md for project overview
Import rules from @./docs/coding-standards.md
Personal: @~/.claude/my-preferences.md
```
- Expanded at session start
- Recursive up to 5 levels deep
- First-time approval dialog for external imports

## 1.4 Size Limits

- **Recommended:** Under 200 lines per file
- **No hard limit** but adherence drops significantly above 200 lines
- HTML comments `<!-- -->` are stripped before loading (zero token cost)

# SECTION 2: MEMORY SYSTEM

## 2.1 Auto-Memory Overview

Claude writes its own notes during sessions, persisting learnings across conversations.

- **Default:** Enabled (v2.1.59+)
- **Toggle:** `/memory` command, `autoMemoryEnabled: false`, or `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`

## 2.2 Storage Location

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # Index (loaded into context, 200 lines max)
├── debugging.md       # Topic files
├── patterns.md
├── api-conventions.md
└── ...
```

Per git repository (not per worktree). Custom location via `autoMemoryDirectory` setting.

## 2.3 Loading Behavior

- **At session start:** First 200 lines / 25KB of MEMORY.md
- **During session:** Topic files read on-demand via Read tool
- **After compaction:** Re-read fresh from disk
- **Claude maintains** the index, splits larger notes into topic files

## 2.4 Memory Types (Freeform)

Build commands, debugging insights, architecture notes, code style, API conventions, reference info, debugging patterns — Claude decides what to save.

## 2.5 Programmatic Access

Memory files are plain markdown. Scripts can read/write. Changes take effect at next session start.

# SECTION 3: SETTINGS.JSON

## 3.1 Four-Level Scope System

| Scope | Location | Priority |
|-------|----------|----------|
| **Managed** | System location | Highest (enforced) |
| **Project** | `.claude/settings.json` | High |
| **Local** | `.claude/settings.local.json` | Medium |
| **User** | `~/.claude/settings.json` | Lowest |

Arrays merge across scopes. Objects merge at top level.

## 3.2 Complete Settings Schema

| Setting | Type | Purpose |

| `effortLevel` | string | Default effort: `low`, `medium`, `high`, `max` |

| `theme` | string | `light`, `dark`, `auto` |

| `autoMemoryDirectory` | string | Custom memory location |
| `agent` | string | Default subagent for main thread |
| `defaultMode` | string | Permission mode |
| `permissions` | object | `allow`, `ask`, `deny`, `additionalDirectories` |
| `sandbox` | object | Filesystem isolation |
| `hooks` | object | Lifecycle hooks |
| `disableAllHooks` | boolean | Disable all hooks |
| `env` | object | Environment variables |
| `enabledPlugins` | array | Plugins to enable |
| `claudeMdExcludes` | array | CLAUDE.md files to skip |
| `cleanupPeriodDays` | number | Session cleanup |
| `vim` | boolean | Vim mode |
| `statusLine` | object | Status line configuration |
| `workingDirectory` | string | Override CWD |

## 3.3 Permissions Configuration

```json
{
  "permissions": {
    "allow": ["Bash(git log *)", "Read", "Edit"],
    "ask": ["Bash(git push *)"],
    "deny": ["WebFetch", "Bash(rm *)"],
    "additionalDirectories": ["../shared/"]
  }
}
```

**Permission rule syntax:**
- `Tool` — all invocations
- `Tool(pattern)` — matching invocations
- `Tool(path/glob)` — file-based matching
- Pipe alternation: `Edit|Write`
- Regex: `mcp__.*__write`

## 3.4 Sandbox Configuration

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["/tmp/build"],
      "denyWrite": ["/etc"],
      "denyRead": ["~/.aws/credentials"]
    }
  }
}
```

# SECTION 4: HOOKS SYSTEM

## 4.1 Overview

27 lifecycle events. 4 hook types (command, http, prompt, agent). See `05-CLAUDE-CODE-HOOKS-SUBAGENTS-TEAMS-RESEARCH.md` for full specification.

## 4.2 Hook Types

### Command hooks
```json
{"type": "command", "command": "script.sh", "async": false, "shell": "bash"}
```
Exit codes: 0=allow, 2=block, other=warn.

### HTTP hooks
```json
{"type": "http", "url": "http://...", "headers": {}, "allowedEnvVars": []}
```

### Prompt hooks
```json
{"type": "prompt", "prompt": "Is this safe?", "model": "fast-model"}
```

### Agent hooks
```json
{"type": "agent", "agent": "validator", "prompt": "Validate this"}
```

## 4.3 Common Patterns

- Auto-format after edits (PostToolUse)
- Block dangerous commands (PreToolUse, exit 2)
- Audit logging (PostToolUse)
- Auto-inject context after compaction (PostCompact)
- Budget checking (PreToolUse for Write)
- Board approval gates (PreToolUse for specific paths)

# SECTION 5: TASKS SYSTEM

## 5.1 What Are Tasks?

Tracking mechanism for multi-step work within a session.

- **TaskCreate** — create new task with title, description, status
- **TaskList** — list all tasks in session
- **TaskUpdate** — update status (pending -> in_progress -> completed)
- **TaskGet** — retrieve specific task

Toggle visibility: `Ctrl+T`. Clear: `/clear`.

Tasks are NOT persistent across sessions. They're for in-session tracking only.

# SECTION 6: CLI FEATURES

## 6.1 All CLI Commands

```bash
claude                          # Interactive session
claude "query"                  # Start with prompt
claude -p "query"               # Print mode (non-interactive)
claude -c                       # Continue most recent
claude -r "name" "query"        # Resume named session
claude --resume                 # Interactive session picker
claude auth login/logout/status # Authentication
claude agents                   # List subagents
claude mcp                      # Configure MCP servers
claude plugin install/uninstall/enable/disable
claude remote-control           # Remote control server
claude --teleport               # Resume web session
claude update                   # Update CLI
claude --version                # Show version
```

## 6.2 Complete CLI Flags

|------|-------------|

## 6.3 Non-Interactive Mode (Print Mode)

```bash
# Basic
claude -p "Summarize this project"

# Continue conversations
session_id=$(claude -p "Start analysis" --output-format json | jq -r '.session_id')
claude -p "Go deeper" --resume "$session_id"

# Structured output
claude -p "Extract functions" --output-format json --json-schema '{...}'

# Stream responses
claude -p "Write poem" --output-format stream-json --verbose

# Auto-approve tools
claude -p "Run tests and fix" --allowedTools "Bash,Read,Edit"
```

## 6.4 Scripting Patterns

### Single query:
```bash
claude -p "Check for errors" --bare --allowedTools "Bash" --output-format json | jq -r '.result'
```

### Multi-step workflow:
```bash
session_id=$(claude -p "Analyze" --output-format json | jq -r '.session_id')
claude -p "Fix issues" --resume "$session_id" --allowedTools "Bash,Edit,Read"
claude -p "Verify fixes" --resume "$session_id" --allowedTools "Bash"
```

### Pipe content:
```bash
cat logs.txt | claude -p "Find the error" --allowedTools "Read"
gh pr diff | claude -p "Review for security"
```

# SECTION 9: CONSTRAINTS & LIMITATIONS

## CLAUDE.md
- Not enforced (guidance, not rules)
- Token cost real (every line)
- No conflict resolution
- Size impact above 200 lines
- Imports are static

## Auto-Memory
- Machine-local (no sync)
- First 200 lines only at startup
- Per-repo (shared across worktrees)
- Not version controlled

## Settings
- No runtime reload (new session needed)
- Bash permission patterns easily bypassed
- No per-session overrides

## Hooks
- No nested hooks
- Timeout limits (600s command, 30s prompt, 60s agent)
- Order not guaranteed (parallel)
- No conditional chaining

## Subagents
- Cannot spawn subagents (no nesting)
- Fresh context (no parent history)
- Communication only via returned results
- Token overhead per spawn

## CLI
- Print mode exits after one turn
- No interactive prompts in print mode
- Sessions are machine-local
- Bare mode skips all auto-discovery

# SECTION 10: BEST PRACTICES

## CLAUDE.md
- Under 200 lines per file
- Specific, testable instructions
- Split using `.claude/rules/`
- Check in to git

## Memory
- Ask Claude to consult before work
- Ask Claude to update after completion
- Periodically trim

## Settings
- Project settings for team-shared rules
- Local settings for personal overrides
- Document permission rules in CLAUDE.md

## Hooks
- Command hooks for quick validation
- Prompt hooks for semantic decisions
- Agent hooks for complex validation
- Log execution in debug mode

## Subagents
- Focused (one task each)
- Clear descriptions
- Restrict tool access
- Use persistent memory for institutional knowledge

## CLI Scripting
- `--bare` for CI/CD
- `--allowedTools` explicitly
- `--output-format json` for parsing
- `--max-turns` and `--max-budget-usd` for safety
