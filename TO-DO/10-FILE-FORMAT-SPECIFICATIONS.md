# File Format Specifications — Every File Type in the System

**Date:** 2026-03-31 (Updated continuously)
**Purpose:** Exact format specifications with examples for every structured file type in the OrgAgent system. These are the canonical formats — all skills, hooks, and GUI parsers must follow these specifications.
**Total Formats:** 26

---

## 1. Organisation Config (`org/config.md`)

Stores all configurable org-level settings. Created by the `/onboard` skill.

```markdown
---
name: "Acme Corp"
language: en
industry: "E-commerce"
created: 2026-03-31
oversight_level: approve-everything
heartbeat_interval: 2h
tone: professional
reporting_frequency: daily
default_agent_model: sonnet
ceo_model: opus
cao_model: opus
manager_model: sonnet
worker_model: haiku
max_budget_per_run_usd: 5.00
---

# Organisation Configuration

This file is auto-generated by the onboarding process.
Do not edit manually unless you know what you're doing.
```

**Fields:**
| Field | Type | Values | Description |
|-------|------|--------|-------------|
| `name` | string | Any | Organisation display name |
| `language` | string | ISO 639-1 code (`en`, `da`, `de`, etc.) | All agent output in this language |
| `industry` | string | Any | Business domain |
| `created` | date | YYYY-MM-DD | Org creation date |
| `oversight_level` | enum | `approve-everything`, `approve-strategy-only`, `hands-off` | Board oversight mode |
| `heartbeat_interval` | duration | `30m`, `1h`, `2h`, etc. | Recommended heartbeat frequency |
| `tone` | enum | `professional`, `casual`, `formal` | Communication style |
| `reporting_frequency` | enum | `daily`, `weekly`, `per-heartbeat` | How often agents write reports |
| `default_agent_model` | string | `opus`, `sonnet`, `haiku` | Default model for new agents |
| `ceo_model` | string | Model ID | CEO's model |
| `cao_model` | string | Model ID | CAO's model |
| `manager_model` | string | Model ID | Default for managers |
| `worker_model` | string | Model ID | Default for workers |
| `max_budget_per_run` | number | Decimal | Max cost per single agent run (in configured currency) |
| `spending_limits.ceo_approval_limit` | number | Decimal | Max amount CEO can approve without board |
| `spending_limits.manager_approval_limit` | number | Decimal | Max amount managers can approve |
| `spending_limits.board_required_above` | number | Decimal | Board approval required above this |
| `n8n_available` | boolean | true/false | Whether n8n is available for integrations |
| `browser_enabled` | boolean | true/false | Whether Playwright browser automation is enabled |

## 2. Organisation Alignment (`org/alignment.md`)

The mission, values, and principles document. Created by `/onboard`, referenced by all agents.

```markdown
# Organisation Alignment — Acme Corp

## Mission
To become the leading sustainable e-commerce platform in Scandinavia,
making eco-friendly products accessible and affordable.

## Vision
A world where every online purchase contributes to environmental sustainability.

## Core Values

1. **Sustainability First** — Every decision considers environmental impact
2. **Customer Obsession** — We exist to serve our customers
3. **Radical Transparency** — We share our processes, costs, and sourcing openly
4. **Speed with Quality** — Move fast but never compromise on product quality
5. **Data-Driven** — Decisions backed by evidence, not assumptions

## Principles

- Always tie work to a strategic initiative — no orphan tasks
- Escalate early — it's better to ask than to assume
- Document decisions — future agents need context
- Respect budget constraints — every API call has a cost
- Communicate clearly — messages should be actionable

## Ethical Boundaries

- Never misrepresent products or services
- Never generate fake reviews or testimonials
- Never scrape competitor data without authorization
- Never store or process customer personal data
- Never make financial commitments without board approval

## Decision-Making Style

- **Speed:** Deliberate — take time for important decisions, move fast on routine ones
- **Risk Tolerance:** Conservative — prefer proven approaches over experimental ones
- **Conflict Resolution:** Escalate to supervisor, then to board if unresolved
```

**Structure rules:**
- Must have: Mission, Vision, Core Values sections
- Optional: Principles, Ethical Boundaries, Decision-Making Style
- Free-form markdown content within each section
- Agents reference this file for value alignment in all decisions

## 3. Organisation Chart (`org/orgchart.md`)

Machine-readable org tree. Updated by CAO on every hire/fire/restructure.

```markdown
# Organisation Chart

> Last updated: 2026-03-31T10:00:00
> Total agents: 5 (3 active, 1 pending, 1 terminated)

- **Board** (human) — Governance & strategic oversight
  - **CEO** (active, @ceo) — Chief Executive Officer
    - **CAO** (active, @cao) — Chief Agents Officer
    - **Marketing Manager** (active, @marketing-manager) — Marketing Department Lead
      - **SEO Agent** (pending-approval, @seo-agent) — SEO Specialist
      - **Social Media Agent** (active, @social-media-agent) — Social Media Specialist
    - **Sales Manager** (terminated, @sales-manager) — Former Sales Department Lead
```

**Parse specification:**

| Element | Format | Example |
|---------|--------|---------|
| Hierarchy | 2-space indentation per level | `  - ` = depth 1, `    - ` = depth 2 |
| Display name | Bold text | `**CEO**` |
| Status | In parentheses | `(active, @ceo)`, `(human)` |
| Agent ID | `@kebab-case` | `@marketing-manager` |
| Role description | After ` — ` (em dash) | `— Chief Executive Officer` |

**Valid statuses:** `human`, `active`, `pending-approval`, `paused`, `terminated`

**Parsing pseudocode:**
```javascript
for each line:
  depth = countLeadingSpaces(line) / 2
  name = extractBold(line)           // between ** **
  status = extractParenStatus(line)  // first word in parens
  agentId = extractAtId(line)        // @xxx pattern
  title = extractAfterEmDash(line)   // after " — "
```

## 4. Task Files (`org/agents/{name}/tasks/{status}/*.md`)

Individual task documents that move between `backlog/`, `active/`, and `done/` directories.

```markdown
---
id: task-20260331-001
title: Create Q2 SEO strategy document
priority: high
status: active
assigned_to: seo-agent
assigned_by: marketing-manager
initiative: q2-marketing-growth
created: 2026-03-31T10:00:00
started: 2026-03-31T10:30:00
completed:
deadline: 2026-04-15
estimated_cost_usd: 0.50
---

## Description

Create a comprehensive SEO strategy for Q2 focusing on organic traffic growth.
Analyze current keyword rankings, identify opportunities, and create a content
calendar optimized for search visibility.

## Acceptance Criteria

- [ ] Current keyword ranking analysis complete
- [ ] Top 20 keyword opportunities identified
- [ ] Content calendar for Q2 drafted (12 weeks)
- [ ] Technical SEO audit checklist created
- [ ] Competitor backlink analysis included

## Context

Ref: org/initiatives/q2-marketing-growth.md
Budget allocated: $0.50 per execution
Reports to: @marketing-manager

## Results

_(filled in by the assigned agent upon completion)_
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Unique ID: `task-YYYYMMDD-NNN` |
| `title` | string | Yes | Human-readable task title |
| `priority` | enum | Yes | `critical`, `high`, `medium`, `low` |
| `status` | enum | Yes | `backlog`, `active`, `done`, `blocked` |
| `assigned_to` | string | Yes | Agent ID (matches @id in orgchart) |
| `assigned_by` | string | Yes | Agent ID of assigner |
| `initiative` | string | Yes | Initiative slug (matches filename in org/initiatives/) |
| `created` | datetime | Yes | ISO 8601 timestamp |
| `started` | datetime | No | When agent began working |
| `completed` | datetime | No | When agent finished |
| `deadline` | date | No | Target completion date |
| `estimated_cost_usd` | number | No | Estimated API cost |

**File naming:** `{id}.md` (e.g., `task-20260331-001.md`)

**Movement rules:**
- Created in `tasks/backlog/` with `status: backlog`
- Agent moves to `tasks/active/` and sets `status: active`, `started: <now>`
- Agent moves to `tasks/done/` and sets `status: done`, `completed: <now>`, fills in Results section
- If blocked: stays in current directory, sets `status: blocked`, adds blocker info

**ID generation:** The assigning agent checks existing files with today's date prefix and increments. Example: if `task-20260331-003.md` exists, next is `task-20260331-004.md`.

## 5. Notification Files (`org/agents/{name}/inbox/*.md`)

Lightweight notifications pointing to thread files. Replaces the old inbox/outbox message system. Full conversations live in `org/threads/` — inbox holds only pointers.

> **ARCHITECTURAL NOTE:** The outbox directory has been ELIMINATED. Thread files (`org/threads/`) are the single source of truth for all conversations. The inbox now holds only lightweight notifications.

```markdown
---
type: thread-notification
thread_id: thread-q2-seo-20260331
thread_path: org/threads/marketing/thread-q2-seo-20260331.md
msg_id: MSG-20260331-100500-ceo
from: ceo
timestamp: 2026-03-31T10:05:00
read: false
subject: "New directive in: Q2 SEO Strategy Kickoff"
---
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | Yes | Always `thread-notification` |
| `thread_id` | string | Yes | ID of the thread containing the message |
| `thread_path` | string | Yes | Full path to the thread file |
| `msg_id` | string | Yes | Greppable message ID within the thread: `MSG-YYYYMMDD-HHMMSS-{from}` |
| `from` | string | Yes | Who sent the message that triggered this notification |
| `timestamp` | datetime | Yes | ISO 8601 |
| `read` | boolean | Yes | `false` when created, `true` when processed |
| `subject` | string | Yes | Brief description of the thread message |

**File naming:** `notif-YYYYMMDD-HHMMSS-{from}.md` (e.g., `notif-20260331-100500-ceo.md`)

**Lifecycle:**
1. When an agent appends a message to a thread, it creates a notification in each participant's `inbox/`
2. Recipient reads during heartbeat, sets `read: true`
3. Recipient then reads the actual thread file for the full message
4. Old notifications (>7 days, read) can be deleted during maintenance

**No outbox.** The thread file is the record of what was sent.

## 6. Approval Files (`org/board/approvals/*.md`)

Board proposals awaiting human decision.

```markdown
---
id: approval-hire-seo-agent-20260331
type: hire
proposed_by: cao
proposed_date: 2026-03-31T10:05:00
status: pending
decided_by:
decided_date:
decision_reason:
---

## Proposal: Hire SEO Agent

### Justification

The Marketing Manager has identified the need for dedicated SEO expertise.
Currently, no agent in the organisation has SEO capabilities. The Q2 marketing
initiative (org/initiatives/q2-marketing-growth.md) requires keyword research,
content optimization, and technical SEO — tasks that need a specialist.

### Proposed Role

- **Name:** seo-agent
- **Title:** SEO Specialist
- **Reports to:** marketing-manager (@marketing-manager)
- **Model:** haiku (cost-efficient for research tasks)
- **Skills:** report, message, escalate
- **Tools:** Read, Write, Edit, Bash, Grep, Glob, WebSearch, WebFetch

### Budget Impact

| Metric | Value |
|--------|-------|
| Current org monthly budget | $500.00 |
| Currently allocated | $350.00 |
| This agent's estimated monthly cost | $30.00 |
| New allocated total | $380.00 |
| Remaining | $120.00 |

### Risk Assessment

- **Low risk:** SEO research is read-heavy, low API cost
- **No external access needed** beyond web search
- **Bounded scope:** Reports only to marketing-manager
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | `approval-{type}-{name}-YYYYMMDD` |
| `type` | enum | Yes | `hire`, `fire`, `reconfigure`, `strategy`, `budget`, `restructure` |
| `proposed_by` | string | Yes | Agent ID of proposer |
| `proposed_date` | datetime | Yes | ISO 8601 |
| `status` | enum | Yes | `pending`, `approved`, `rejected` |
| `decided_by` | string | No | `board` or agent ID (if delegated) |
| `decided_date` | datetime | No | When decision was made |
| `decision_reason` | string | No | Reason (especially for rejections) |

**File naming:** `{id}.md`

**State transitions:**
```
pending → approved → (file moves to org/board/decisions/)
pending → rejected → (file moves to org/board/decisions/)
```

# Audit Log

**Columns:**
| Column | Description |
|--------|-------------|
| Timestamp | ISO 8601 datetime |
| Agent | Agent ID, `board` for human, `SYSTEM` for automated |
| Action | Verb describing the action (see action types below) |
| Target | File path (relative to org/) or `—` if not applicable |
| Details | Human-readable description |

**Action types:**
`org-created`, `agent-created`, `agent-terminated`, `agent-reconfigured`, `heartbeat-start`, `heartbeat-end`, `task-create`, `task-start`, `task-complete`, `task-block`, `message-send`, `hire-propose`, `fire-propose`, `approval`, `rejection`, `budget-update`, `error`

**Append format:** Each hook invocation appends exactly one line:
```
| {timestamp} | {agent} | {action} | {target} | {details} |
```

# Soul

You are a strategic marketing leader. You think in campaigns, not tasks.

- Agents cannot modify their own SOUL.md

## 12. Agent INSTRUCTIONS.md (`org/agents/{name}/INSTRUCTIONS.md`)

Operating manual — HOW the agent works.

```markdown
# Instructions

## Context Loading
At the start of every session, read these files:
1. `org/alignment.md` — organisation mission and values
2. `org/config.md` — org configuration (especially language setting)
3. `org/agents/marketing-manager/SOUL.md` — your behavioral identity
4. `org/agents/marketing-manager/IDENTITY.md` — your role and status
5. `org/agents/marketing-manager/MEMORY.md` — your persistent knowledge
6. `org/orgchart.md` — current org structure

## Your Reports
- **You report to:** CEO (@ceo)
- **Reports to you:** SEO Agent (@seo-agent), Social Media Agent (@social-media-agent)

## Operating Procedures

### Inbox Processing
1. Read all unread messages in `org/agents/marketing-manager/inbox/`
2. For each message, determine action: respond, delegate, escalate, or acknowledge
3. Mark processed messages as `read: true`

### Task Management
1. Check `tasks/backlog/` for new assignments
2. If you can delegate to a subordinate, create a task in their `tasks/backlog/` and notify via their inbox
3. If you must do it yourself, move to `tasks/active/` and begin work
4. Write results in the task file's Results section, move to `tasks/done/`

### Delegation Rules
- Only delegate to agents that report to you (check orgchart)
- Always create a task file in the subordinate's backlog
- Always send a notification message to their inbox
- Set realistic deadlines based on task complexity
- Check subordinate workload before assigning (count their active tasks)

### Reporting
- Write daily status reports to `reports/daily-{YYYY-MM-DD}.md`
- Include: tasks completed, tasks in progress, blockers, budget usage
- Report should be concise (under 50 lines)

### Escalation
- Escalate to CEO when: budget concerns, cross-department conflicts, strategic questions
- Use the /escalate skill or write to `org/agents/ceo/inbox/`

## Constraints
- NEVER act outside your department scope (marketing only)
- NEVER exceed your budget allocation
- NEVER modify agent definitions (only CAO can do this)
- NEVER skip-level delegate (don't assign tasks to another department's workers)
- NEVER modify your own SOUL.md or IDENTITY.md
- During heartbeats: do NOT use the Agent tool to spawn subagents
- ALL content must be written in the language specified in org/config.md
```

## 13. Agent HEARTBEAT.md (`org/agents/{name}/HEARTBEAT.md`)

Periodic checklist — WHAT the agent does on each heartbeat cycle.

```markdown
# Heartbeat Checklist

Every heartbeat cycle, review and act on these items in order:

1. **Check urgent messages** — Read `org/messages/urgent/` for org-wide alerts
2. **Process inbox** — Read all unread messages in your `inbox/`, respond or delegate
3. **Check approvals** — If you have pending proposals, check `org/board/approvals/` for decisions
4. **Review active tasks** — Check `tasks/active/` for work in progress, continue or complete
5. **Process backlog** — Check `tasks/backlog/` for new assignments, prioritize and begin or delegate
6. **Review subordinates** — Check direct reports' latest reports in `org/agents/*/reports/`
7. **Budget check** — Verify your department's budget status in `org/budgets/overview.md`
8. **Write status report** — Summarize this heartbeat's actions to `reports/daily-{YYYY-MM-DD}.md`
9. **Update memory** — Log important decisions and learnings to `memory/{YYYY-MM-DD}.md`
```
